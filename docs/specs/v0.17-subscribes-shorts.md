# v0.17 — Shorts-aware subscribes

Status: **draft, pending sign-off**
Target version: **v0.17.0**

## Problem

`subscribes update` for YouTube uses the channel RSS feed
(`feeds/videos.xml?channel_id=...`) by default. **YouTube RSS does not
include Shorts** — only full uploads. Two consequences:

1. A channel that publishes mostly Shorts looks "silent" through
   `subscribes update`, even when there's plenty of fresh content.
2. A channel that paused long-form uploads but is actively Shorts-only
   for the requested window (`--days 7`) returns nothing — the user
   misses what's actually new on the channel.

The `--no-rss` escape hatch exists, but it (a) is opt-in per-call,
(b) doesn't differentiate between Shorts and full videos, and (c) is
slower because it scrapes the whole channel page.

## Goals

- Make `subscribes update` aware of Shorts as a first-class content type.
- Default behavior: don't miss Shorts when there are no fresh full
  videos in the requested window.
- Per-channel preference: remember "this channel is mostly Shorts" so
  the right mode is used without flags every time.
- Per-call override: ad-hoc switch (`--shorts-only`, etc.) without
  changing stored preferences.
- Cap: don't silently transcribe 30 Shorts at once. Cap default + warn
  when the cap is hit.

## Non-goals (this version)

- Instagram Reels / TikTok analogue routing. TikTok is effectively all
  short-form, and IG Reels are already covered by the same flat scrape
  used for IG posts — there's no separate `/shorts` tab to add.
  Revisit only on explicit request.
- Interactive "found N shorts, take all / first M / none?" prompt.
  Cap + stderr warning covers the 80% case; a TTY prompt can land in
  a follow-up.
- Per-channel `last_seen_published_shorts` split. One cursor stays
  authoritative for the channel; we dedup mixed streams by `video_id`.
- Differentiating Shorts by `duration_sec <= 60`. We classify by what
  YouTube classifies — anything returned by the channel's `/shorts`
  tab is a Short; everything else is a full video. (yt-dlp returns
  shorts URLs in the form `https://www.youtube.com/shorts/<id>`, which
  is also how we mark them in batch metadata.)

## Mode taxonomy

Per-subscription `mode` field with four values:

| Mode                | Sources fetched                           | Window semantics                              |
| ------------------- | ----------------------------------------- | --------------------------------------------- |
| `auto` (default)    | RSS (full videos). Fallback: `/shorts` IF RSS returned 0 entries in the window. | Standard window (incremental cursor / `--days N`) applied to whichever stream produced entries. |
| `videos-only`       | RSS only. No fallback.                    | Standard window. Current pre-v0.17 behavior. |
| `shorts-only`       | yt-dlp `/shorts` scrape only.             | Standard window.                              |
| `shorts-and-videos` | BOTH RSS and `/shorts` always, deduped by `video_id`, sorted by `published_at desc`. No fallback logic — explicitly "I want everything." | Standard window over the merged stream. |

Default for new subscriptions: `auto`.
Default for existing subscriptions (loaded from pre-v0.17 toml without
the `mode` field): `auto`. Backward-compat handled in `store._from_dict`
the same way `platform` was handled in v0.8.

### Per-call override

On `subscribes update` (all subscriptions in scope):

| Flag                          | Effective mode per channel for this call only |
| ----------------------------- | --------------------------------------------- |
| `--shorts-only`               | `shorts-only` (overrides stored mode)         |
| `--include-shorts`            | `shorts-and-videos` (overrides stored mode)   |
| `--no-shorts`                 | `videos-only` (overrides stored mode)         |
| *(none)*                      | Stored per-subscription mode                  |

Mutex group — at most one of the three at a time. Click `cls=Mutex` or
manual check raises a clear error.

### Cap

- `cfg.shorts_max_per_update: int = 5` (new Config field, persisted to
  `~/.neurolearn/config.toml`).
- Flag `--shorts-cap N` on `subscribes update` overrides for one call.
  `N=0` means "no cap" (take everything in window).
- When the cap fires, stderr warns:
  `[warn] @handle: 17 shorts found in window, taking 5 newest. Raise with --shorts-cap N or set in config.toml.`
- Cap applies **per-channel per-update**, not globally.
- Cap is applied **after** window filtering, sort by `published_at desc`,
  before passing to the candidate list.
- Cap applies **only to Shorts**, not full videos. Full videos remain
  uncapped (channels rarely fire-hose them anyway, and a fresh-uploads
  cap would be a behavior change for `videos-only` users who never
  opted into this feature).

## CLI surface

### `subscribes add`

New option:
```
--mode [auto|videos-only|shorts-only|shorts-and-videos]   default: auto
```

For YouTube channels only — silently ignored (but accepted) for IG/TT
to avoid breaking command syntax in scripts. We may surface a tiny
"mode currently only applies to YouTube" hint in interactive mode; in
non-TTY we just no-op.

### `subscribes set-mode` (new)

```
neurolearn subscribes set-mode <identifier> <mode>
```
Where `<identifier>` is handle / URL / channel_id (same matcher as
`remove`). Updates the stored `mode` field, preserves comments via
tomlkit. Errors out with exit 3 if channel not found.

### `subscribes update`

Three new flags:
```
--shorts-only          # mutex with --include-shorts / --no-shorts
--include-shorts       # mutex with --shorts-only / --no-shorts
--no-shorts            # mutex with --shorts-only / --include-shorts
--shorts-cap N         # 0 = no cap
```

### `subscribes list`

New column **Mode** between **Group** and **Channel ID**.
Only meaningfully populated for YouTube rows; for IG/TT rows shown
as `—` (mode has no effect on those platforms in v0.17).

## Pipeline routing (decision tree)

```
for ch in channels (after group/platform filter):
    effective_mode = cli_override_mode or ch.mode  # ch.mode defaults to "auto"
    if ch.platform != "youtube":
        # IG / TT — unchanged, mode is ignored
        entries = existing_fetch_path(ch)
    else:
        match effective_mode:
            case "videos-only":
                entries = fetch_rss_or_ytdlp_videos(ch)
            case "shorts-only":
                entries = fetch_shorts(ch, cap=...)
            case "shorts-and-videos":
                videos = fetch_rss_or_ytdlp_videos(ch)
                shorts = fetch_shorts(ch, cap=...)
                entries = dedup_by_id(videos + shorts)
                entries.sort(key=published_at, reverse=True)
            case "auto":
                videos = fetch_rss_or_ytdlp_videos(ch)
                videos_in_window = apply_window(videos, ...)
                if videos_in_window:
                    entries = videos_in_window
                else:
                    entries = fetch_shorts(ch, cap=...)

    entries = apply_window(entries, ...)
    # rest unchanged: candidates, screening, batch, state update
```

Notes:
- `auto` is the **only** mode that does window-first-then-fallback.
  All other modes fetch both/either streams up front.
- `fetch_shorts` does yt-dlp `expand_channel_or_playlist(ch.url + "/shorts", limit=...)`.
  Returns `_ChannelVideo` entries with `duration_sec` populated (yt-dlp
  always returns it for `/shorts` results) and URLs normalised to the
  `youtube.com/shorts/<id>` form (yt-dlp does this naturally).
- Dedup by `video_id`. If the same id appears in both streams (rare —
  YouTube doesn't usually surface a video on both `/videos` and
  `/shorts` tabs, but defensive against future tab schema changes),
  prefer the Short entry (it has `duration_sec`, RSS doesn't).
- Cap applies **inside** `fetch_shorts` after window filter, before
  return. The warning is emitted there as well.

## Backward compatibility

- `subscribes.toml` entries without `mode` → `mode="auto"` at load time.
  This means existing users start getting Shorts fallback **automatically**
  on first v0.17 `subscribes update`. Surfacing this in CHANGELOG and
  the upgrade notes is mandatory: it's behavior change for
  zero-mode-set channels. Users who want the old behavior set
  `subscribes set-mode <ch> videos-only`.
- `--no-rss` flag stays. Behavior: when set, the "RSS or yt-dlp" choice
  inside `fetch_rss_or_ytdlp_videos` picks yt-dlp. The `/shorts`
  fetcher always uses yt-dlp regardless of `--no-rss`.
- `last_seen_published` / `last_seen_video_id` cursors remain
  single-channel; mixed mode advances based on the **newest** entry
  across both streams. Documented as such — a mid-window switch from
  `shorts-only` to `videos-only` will not "rewind" the cursor, which
  is consistent with how `--days` overrides also work today.
- `cfg.shorts_max_per_update` is a new field. Loaded with default `5`
  on configs without the key (standard `getattr(…, default)` pattern
  used throughout `config.py`).

## Cost / quota considerations

- Worst case on `auto`: channel produces nothing in window → one extra
  yt-dlp scrape per channel per update. With 50 subscriptions all
  dormant → 50 extra `/shorts` scrapes. Roughly 1-2s each on warm
  network. Acceptable.
- Cap default of 5 keeps Groq audio transcription cost predictable:
  ~5 × 60s = 5 minutes of audio per channel max, well within free-tier
  Whisper limits.
- `--learn-into <memory>` blowup risk: 5 shorts × 10 dormant channels
  = 50 candidates briefing manifest. Existing system handles that;
  not introducing additional safeguards in v0.17.

## Known follow-ups (deferred)

1. TTY prompt when cap fires: "23 shorts found — take all / first 5 /
   none?". Useful but not blocking.
2. Per-channel cap override in `subscribes.toml`
   (`mode = "shorts-only", shorts_cap = 10`). Probably needed once
   users start tuning per-channel. v0.18 candidate.
3. Suppress "RSS empty → /shorts also empty" repeat scrapes: cache
   "last attempted empty fetch at" timestamp and skip the fallback for
   N hours if both came back empty recently. Optimization, not
   correctness.
4. IG Reels / TikTok mode parity if a real user need appears.

## Risks / open questions

1. **yt-dlp behavior on `<channel>/shorts` URL**: confirmed by
   experimentation before code. Expected: same flat-extract pipe as
   `<channel>` or `<channel>/videos`; entries get `duration` and
   `webpage_url` with `/shorts/` segment. If yt-dlp's flat extract
   doesn't honor the tab segment, fallback plan: use the channel's
   `/shorts` URL through the regular extract and filter by
   `webpage_url contains '/shorts/'`. Will resolve during implementation.
2. **State advancement on `auto` fallback**: if RSS was empty and we
   fell back to `/shorts`, do we advance `last_seen_published` to the
   newest Short? **Yes** — same rule as today ("advance to newest
   thing we successfully discovered, regardless of transcription
   outcome"). Documented; no special-case logic.
3. **Channel-resolver caching of `/shorts` URLs**: `subscribes add`
   accepts `youtube.com/shorts/<id>` URLs (v0.16.0). They resolve to
   channel URL → fine, no change needed.

## Test plan (summary)

Acceptance:

- `mode=auto`, RSS returns entries in window → shorts NOT fetched.
- `mode=auto`, RSS returns zero in window → shorts fetched, capped.
- `mode=videos-only`, RSS empty → no shorts fetched. (Regression of
  current behavior.)
- `mode=shorts-only` → only `/shorts` scrape, RSS not called.
- `mode=shorts-and-videos` → both fetched, dedup verified by
  injecting the same id into both mocks.
- CLI override `--no-shorts` on a `shorts-only`-stored channel →
  effectively `videos-only` for that one call.
- CLI mutex check: `--shorts-only --no-shorts` → exit 2 with error.
- Cap: 17 shorts in window, `--shorts-cap 5` → 5 candidates, stderr
  warning includes `17` and `5`.
- Cap=0 means "no cap": 17 shorts → 17 candidates, no warning.
- Backward compat: loading pre-v0.17 `subscribes.toml` (no `mode`
  field) → all channels behave as `auto`.
- `subscribes set-mode <unknown>` → exit 3.
- `subscribes list` shows Mode column; YouTube rows populated, IG/TT
  rows show `—`.

Existing test suite (1346 passed) must remain green. Notably:
- `test_first_run_requires_window` — unchanged (window required
  regardless of mode).
- `test_stateful_default_uses_last_seen` — passes through `auto` mode
  which behaves identically to current default when RSS has entries.
